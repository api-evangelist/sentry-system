<p>Native crashes on Android have always been harder to debug than they should be.</p>
<p>The platform has its own crash reporter (<code>debuggerd</code>) that captures the crashing thread, every other running thread, register state, and memory maps into a file called a tombstone. Tombstones have been a part of Android for a long time; in fact, they've been there in one form or another since Android's first commit.</p>
<p>The problem: for most of Android's life, you couldn't read tombstones programmatically from inside your app. That left SDK-based native crash reporting (like ours) stuck replicating infrastructure the platform already had — at the cost of binary overhead, incomplete Java frame symbolication, and a C++ fork we had to maintain against a moving AOSP target.</p>
<p>Android 11 (SDK level 30) introduced <a href="https://developer.android.com/reference/android/app/ApplicationExitInfo"><code>ApplicationExitInfo</code></a>. Android 12 (SDK level 31) added access to the trace input stream for <a href="https://developer.android.com/reference/android/app/ApplicationExitInfo#REASON_CRASH_NATIVE"><code>ApplicationExitInfo.REASON_CRASH_NATIVE</code></a>.</p>
<p>Sentry's Android SDK, as of version <code>8.30.0</code>, reads that stream on all devices running Android 12 and above and ships it as a native crash event. This dramatically improves crash reporting for Android apps that use native code, whether your team wants just basic crash alerts or deep debugging info.</p>
<p>Let's dive into how things worked before, what it took to wire it into the SDK without breaking the existing NDK integration, and the improvements these changes bring.</p>
<h2>Before tombstones: a fork chasing a moving target</h2>
<p>Before tombstone support, the Native SDK (<code>sentry-native</code>) was used in the Android SDK as the primary native error reporting source. Since Android is based on Linux, a considerable part of the SDK could be reused. For those parts that couldn't, work on integrating Android-specific code began in 2019.</p>
<p>Specifically, the integration of <code>libunwindstack</code> (the AOSP platform unwinder still used today to produce stack traces for <code>debuggerd</code> and, in turn, tombstones) was a key moment for supporting native crashes in Sentry's Android SDK.</p>
<p>Why, you ask? Because the Native Development Kit (NDK) did not offer a general-purpose stack walker (narrator: it still does not).</p>
<p><code>libunwindstack</code> is not part of the NDK, but part of the <a href="https://android.googlesource.com/platform/system/unwinding/">Android Open Source Project (AOSP) platform code</a> and thus is not directly accessible to app developers via the usual means. Sentry forked a repository that patched the platform code to build with the NDK, and since then, maintained that fork without any changes in the upstream patched version. This provided stack tracing capabilities inside the rather complicated Android Runtime (ART) environment, which has mixed stack-traces between classic native code, native code that is part of the VM execution, and Java/Kotlin frames that also appear as native frames, since they are either interpreted, JITed, or AOTed.</p>
<p>While this can already be challenging for a normal stack-walker, it is also a problem from the perspective of symbolication: there are more OEM builds than Sentry can realistically collect platform binaries from. So, while a core set of libraries will likely exist in our backend stores, we cannot rely on all of them being available. Thus, symbolication on Android happens on the client-side.</p>
<p>Considerable restructuring in the platform code, however, made manual upstream alignment very hard over time. In addition to that <code>libunwindstack</code> is a C++ library, which means, while being light on standard library usage, it still needs to be linked against it statically in order to ensure being isolated at runtime from ABI-incompatible versions of the standard library.</p>
<p>It also introduced a couple of challenges:</p>
<ul>
<li><p><strong>The biggest issue always was size:</strong> since we must package binaries for <code>x86</code>, <code>x86_64</code>, <code>armeabi-v7a</code>, and <code>arm64-v8a</code>, we currently add around 1MiB of stripped binary to every app that needs native error reporting or instrumentation. The Sentry SDK code only accounts for 20% of that size; the rest is libunwindstack and the C++ infra it depends on.</p>
</li>
<li><p><strong>Incomplete implementation:</strong> since the library size is already significant, certain features have been excluded from the build: there is currently no DEX/OAT symbolication (meaning none of the Java frames are symbolicated), and there is incomplete support for locating DWARF CFI in OAT frames, which often leads to dramatically shortened stack traces in release builds.</p>
</li>
<li><p><strong>Limited context:</strong> Since the <code>inproc</code> backend, which handles the crashes on Android, doesn't stop any threads by design, it also only provides the stack trace of the crashed thread, which, in particular on Android, is often way too little context to uncover the root-cause of a crash</p>
</li>
</ul>
<p>So while all of these are fixable, the effort required is significant and would also lead to a long-term commitment to maintain against the moving target that is AOSP. Introducing tombstone support allows us to fix all the issues mentioned for users who run on Android 12+, which is a significantly growing portion of the incoming events and user base. At the same time, it opens the door to work on better solutions for edge cases.</p>

<h2>What tombstones give us</h2>
<p>The problems outlined above: size, incomplete traces, missing Java symbolication, and maintenance burden, all stem from replicating the platform crash infrastructure that already exists on the device.</p>
<p>Tombstones fix each of them:</p>
<ul>
<li><p><strong>All threads, fully symbolicated.</strong> Where the <code>inproc</code> backend could only capture the crashing thread, tombstones provide stack traces and register sets for every thread at the moment of the crash. On Android, the crashing thread is often just the victim of a problem that originated in another thread. Seeing all of them is the difference between a solvable crash and an enigma.</p>
</li>
<li><p><strong>Java/Kotlin frames resolved.</strong> The platform unwinder has full access to ART internals that a forked NDK build cannot have. DEX/OAT symbolication, which we deliberately excluded to limit binary size, comes for free.</p>
</li>
<li><p><strong>No binary overhead for stack traces.</strong> Tombstones are produced by the platform's own <code>libunwindstack</code>, the same library we have been forking and shipping. The ~1MiB binary weight for all supported ABIs drops to zero for apps that rely on tombstones alone.</p>
</li>
<li><p><strong>Maintenance shifted to the platform.</strong> We consume structured output instead of tracking AOSP restructuring and keeping a C++ fork buildable against the NDK.</p>
</li>
<li><p><strong>Register memory context.</strong> Memory dumps around pointer values in the crashing thread's registers show the data being operated on at the point of the crash. (Not yet integrated into the Sentry event payload or UI.)</p>
</li>
<li><p><strong>Symbol resolution:</strong> Since we now have modules and resolved symbols on the client, we can also strip non-actionable trace contents like runtime-internal frames before sending.</p>
</li>
</ul>
<h2>How to use it</h2>
<p>Tombstone support is available since version <code>8.30.0</code> of <code>sentry-android-core</code>.</p>
<p>If your app runs on Android 12+ and you enable tombstones, you will automatically get more complete reports delivered for all native crashes that affect your app. If you used Native SDK/NDK integration, you will automatically get better stack traces for all your threads and still see the context you created on the native side.</p>
<p>If you have never used the Native SDK interfaces in your native code directly, you can evaluate your options for disabling the NDK integration. If enough users of an app moved on to Android 12+, there is no further use in running both integrations.</p>
<p>If, however, the Native SDK interface is still in direct use, both integrations work together without any visible degradation in user experience.</p>
<p>If you want to turn on the feature, you can do so programmatically via <code>SentryAndroidOptions</code>:</p>
<pre><code class="language-kotlin">SentryAndroid.init(context) { options -&gt;
    options.isTombstoneEnabled = true
}
</code></pre>
<p>Or declaratively in your <code>AndroidManifest.xml</code>:</p>
<pre><code class="language-xml">&lt;meta-data android:name=&quot;io.sentry.tombstone.enable&quot; android:value=&quot;true&quot; /&gt;
</code></pre>
<p>Since tombstones capture every thread at the moment of the crash, you can inspect any of them directly in the issue detail view:</p>

<h2>The implementation challenges</h2>
<p>Tombstone support touched on many layers of the SDK because native crash reporting intersects with session management, event deduplication, envelope caching, event enrichment, and the existing NDK integration that already handles the same class of crashes through a completely different mechanism.</p>
<h3>Sharing infrastructure with ANR detection</h3>
<p>The most immediate architectural challenge was that the SDK already had an integration consuming <code>ApplicationExitInfo</code>: the <code>ANR</code> integration, which handles <code>REASON_ANR</code>. Both integrations need the same lifecycle: query the historical exit list, skip already-reported entries, distinguish the latest (enrichable) entry from older (historical) ones, persist a &quot;last reported&quot; timestamp marker, wait for the previous session to flush, and block until the event is written to disk.</p>
<p>Duplicating this would have been the faster path, but the implementation instead extracted a generic history dispatcher parameterized by a policy interface. Each integration implements the policy (target reason, historical flag, report builder), while the dispatcher owns traversal, ordering, deduplication, and flush coordination. The envelope cache's timestamp marker system was similarly generalized so both ANR and tombstone markers are handled polymorphically.</p>
<p>This refactoring had a cascading consequence for event processing. The existing ANR event processor was tightly coupled to <code>ANR</code> assumptions and enriched every &quot;backfillable&quot; event (which are events that don't have access to the live scope of the session they emerge from, but the scope can forensically be reconstructed) as though it were an <code>ANR</code>. With tombstones now also flowing through as backfillable events, the processor was generalized with an enrichment strategy interface. <code>ANR</code>-specific logic (exception synthesis from textual thread dumps, background/foreground fingerprinting, profile-based culprit identification) moved into a dedicated enricher. At the same time, the shared path (scope backfilling, options backfilling, device/OS context) became the generic default that tombstones are fully served by without needing their own enricher.</p>
<p>A considerable part of this new infrastructure can now be reused for other <code>ApplicationExitInfo</code> categories, likely even when the resulting artifacts won't be events (but rather entries in SDK client reports).</p>
<h3>Coexisting with the NDK integration</h3>
<p>The deeper problem was that tombstones and the existing Sentry NDK integration (using <a href="https://github.com/getsentry/sentry-native/blob/master/ndk/README.md"><code>sentry-native-ndk</code></a>) report the same crash. The Native SDK catches the signal at runtime via its own signal handler and writes an envelope to the &quot;outbox&quot;. The tombstone is generated by the platform's <code>debuggerd</code>, which is invoked after the Native SDK's signal handler chains to the previous handler, but the tombstone only arrives through <code>ApplicationExitInfo</code> on the next launch, after the process has been killed.</p>
<p>If both integrations are active, every native crash produces a duplicate. We need both to get the full picture: the richer stack traces, thread coverage, and up-to-date memory maps from the tombstone, combined with user-supplied scope data from the Native SDK. So we can't simply turn one off in favor of the other.</p>
<p>Solving this required correlating the two events by timestamp (within a 5-second tolerance) and merging them into one. The correlation itself was trivial. The complexity came from the different paths the two events take before they can be merged.</p>
<p>The Native SDK serializes envelopes to a shared app directory (the &quot;outbox&quot;), which acts as a signal to the Android SDK that an envelope is ready to send. For a native crash, this signal arrives too late to be picked up by the normal outbox sending infrastructure during the crash. So, on the next start, that infrastructure loads every envelope fully into memory, because its sole purpose is to send them to the backend. If we reused it for merge discovery, we would deserialize every queued envelope into memory just to find the one native crash event worth merging. On a device that has been offline and accumulated envelopes, this means a spike in memory pressure and CPU load for what is almost always a single match.</p>
<p>Instead, a lightweight scan phase streams through each envelope file, parsing only item headers and extracting the platform and timestamp fields via streaming JSON, without deserializing the full event. A bounded input stream tracks position within each envelope item and skips unread bytes to correctly advance to the next item. Full deserialization only happens once a timestamp match is found. The resulting streaming envelope/event parsing infrastructure can likely be reused in other parts of the SDK.</p>
<p>The merged event carries a <code>TombstoneMerged</code> exception mechanism (alongside the existing <code>Tombstone</code> and <code>signalhandler</code> mechanisms) so the backend, developers, and customers can distinguish provenance.</p>
<h3>Session and <code>crashedLastRun</code> lifecycle</h3>
<p>Native crash reporting interacts with session tracking in ways that require careful coordination. When the tombstone integration processes a crash, it needs to end the previous session as crashed and set <code>crashedLastRun</code> to <code>true</code>. But the NDK integration has its own mechanism for this: a crash marker file checked by the session finalizer on next launch.</p>
<p>A dedicated marker hint (deliberately distinct from the one used for ANRs) was introduced so that the envelope cache can recognize tombstone events during session persistence: when it sees the hint, it ends the previous session as crashed with the crash timestamp. The session finalizer then detects the already-crashed state and sets <code>crashedLastRun</code> accordingly, without re-processing the NDK crash marker. Crucially, the native crash marker file is still cleaned up regardless of whether the tombstone integration handled the crash. Otherwise, the NDK path would re-report it on every subsequent launch.</p>
<h3>The protobuf dependency problem</h3>
<p>Android tombstones use a protobuf format defined in AOSP (<a href="https://cs.android.com/android/platform/superproject/main/+/main:system/core/debuggerd/proto/tombstone.proto"><code>tombstone.proto</code></a>). The initial implementation used <code>protobuf-javalite</code> for decoding, which immediately caused version conflicts for SDK consumers already using protobuf (usually via Firebase). Within a month of the initial release, we replaced it with <code>epitaph</code>, a handwritten decoder for the tombstone protobuf encoding, free of transitive dependencies and weighing around 30KiB. We also added a scheduled CI workflow to monitor AOSP for changes to the tombstone protobuf schema, so we know early if any consequential format changes land in the platform.</p>
<p>The unifying theme across these challenges is that native crash reporting is not a self-contained feature. It sits at the intersection of the SDK's event pipeline, session lifecycle, disk caching, and the existing NDK integration, each of which had been designed with the assumption that it was the only actor in its domain.</p>
<p>Adding tombstone support meant teaching these components to share: the history dispatcher with <code>ANR</code> detection, the outbox with the <code>NDK</code> integration, the session finalizer with a new crash source, and the event processor with a new category of event. We chose refactoring over duplication at each of these intersection points, which made the initial PRs larger and the review cycles a bit longer, but left the architecture at least as clean as we found it. Especially the common Java SDK core did not see any behavioral changes.</p>
<h2>Closing the gap</h2>
<p>Tombstone support closes a gap that has existed since Sentry first shipped native crash reporting on Android: the difference between what the platform knows about a crash and what the SDK could tell you.</p>
<p>While that gap might seem arbitrary since we could replicate parts of the platform's own crash infrastructure inside the app, it only happened by paying the cost of binary size, maintenance burden, and still incomplete results. With <code>ApplicationExitInfo</code> providing programmatic access to the same data that <code>debuggerd</code> produces, we can now offer richer crash context with less overhead and fewer moving parts.</p>
<p>Of course, the limitation is real: this only works on Android 12 and above. For older devices and apps that need instrumentation of their native code beyond error reporting, the NDK integration remains available, and the two coexist cleanly. But with Android 12+ now representing 75% (according to <a href="https://apilevels.com/">apilevels.com</a>, as of 03/2026) of cumulative usage distribution, the balance has tipped. For most apps, tombstone support is the primary native crash reporting path today, and <code>sentry-native-ndk</code> is the fallback.</p>
<p>Tombstone support is available in <code>sentry-android-core</code> 8.30.0 and above. See the <a href="https://docs.sentry.io/platforms/android/">Android SDK docs</a> for configuration details and guidance on whether to keep or drop the NDK integration for your app.</p>
